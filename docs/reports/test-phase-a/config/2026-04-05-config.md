### B. `validateConfig.ts`

#### 実際に検証していること:
- ✅ ~40数値パスの非ゼロ（ゲームパラメータ）
- ✅ 2つの enum フィールド: bridgeTargetCellType, relayPointPlacementCellType
- ✅ アイテム: 全14キー存在、allowedTargetCellTypes 内の有効 CellType、duration≠0、mine_remover/erosion_pause/purify の effectRef 有効性
- ✤ スキル: 全14キー存在、有効 SkillType
- ✅ ステージ: 重複 stageId/floorNumber なし、floor→stage 参照整合性、全ステージが参照されている
- ✅ リワード: 非空プール、有効 kind/weight/minMax、ID 解決

#### 検証していない重要なギャップ:

| カテゴリ | 未検証項目 | 件数 |
|---|---|---|
| 範囲制約 | minWidth≤maxWidth, baseSlots≤maxSlots, 確率/率境界 (0,1] | 8+ |
| 数式文字列 | sizeFormula, intervalFormula などが有効/ホワイトリスト化 | 9 フィールド |
| アイテム意味論 | targeting.mode, effectKind, 真偽値, maxStack、アイテム固有不変条件×14 | 30+ |
| スキル意味論 | rarity, stackLimit, valueRoll.min≤max、スキル固有（chord） | 10+ |
| ステージ幾何学 | 座標が整数/非負/範囲内、hole/CP/spawn の重複なし | 8+ |
| リワード意味論 | プール内重複 ID、levelUp.optionCount>0 | 3 |
| クロスファイル | duration 整合性（dash↔dashDurationMs など）、mineRemoverRefs 解決 | 6+ |
| 構造的安全性 | 誤った型 → ConfigValidationError（生 TypeError ではない） | 全配列/オブジェクトパス |
